1. Опис реалізованих бізнес-сценаріїв

У рамках системи управління бібліотекою було виділено та реалізовано 4 фундаментальні бізнес-сценарії:

Реєстрація користувачів: * Логіка: Система приймає ім'я нового читача, перевіряє його на валідність (не порожнє), створює сутність User і зберігає її в репозиторії. Повертає унікальний ідентифікатор користувача.

Пошук книги:

Логіка: Користувач може знайти книгу за назвою. Сервіс звертається до репозиторію, шукає збіги та формує безпечний об'єкт BookDto, який містить лише необхідну інформацію (включно з текстовим статусом "Доступна" або "Позичена"), приховуючи внутрішню структуру БД.

Кредитування (видача) книги:

Логіка: Перевіряється наявність книги в базі, її поточний статус (чи не позичена вона вже) та існування користувача. Якщо всі умови виконані, статус книги змінюється на IsBorrowed = true, встановлюється BorrowerId, і запис оновлюється в репозиторії.

Повернення книги:

Логіка: Перевіряється наявність книги та факт того, що вона дійсно була видана. Після цього статус книги скидається (IsBorrowed = false, BorrowerId = null), і репозиторій фіксує зміни.

2. Структура коду (Архітектура)

Проєкт організовано за класичним шаблоном Controller-Service-Repository, що забезпечує чітке розділення відповідальності (Separation of Concerns).

LibrarySystem/
--- LibrarySystem.Core/
------ Controllers/   # Точка входу (LibraryController). Приймає запити і викликає сервіси.
------ Services/      # Бізнес-логіка (LibraryService). Реалізує всі перевірки та сценарії.
------ Repositories/  # Доступ до даних (BookRepository, UserRepository). Лише CRUD операції.
------ Models/        # Сутності бази даних (Book, User).
------ DTO/           # Об'єкти передачі даних (BookDto).
--- LibrarySystem.Tests/
------ LibraryServiceTests.cs # Юніт-тести з використанням Moq та xUnit.

3. Рефакторинг і "запахи коду" (Code Smells)

Під час розробки було проведено рефакторинг та дотримано принципів чистого коду:

Усунення "God Object": Замість одного великого класу, який робить усе, логіка розділена на Контролер, Сервіс та Репозиторій.

Впровадження залежностей (Dependency Injection): LibraryService не створює екземпляри репозиторіїв через new, а приймає абстракції (IBookRepository, IUserRepository) через конструктор. Це дозволило легко написати ізольовані модульні тести.

Стандартизація іменування: Тести названі за стандартом MethodName_StateUnderTest_ExpectedBehavior (наприклад, IssueBook_AlreadyBorrowed_ThrowsException), що робить їх самодокументованими.

Безпека даних (DTO): Назовні (через метод FindBookByTitle) передається не модель бази даних Book, а BookDto, що приховує чутливі або службові поля.

4. Інструкція із запуску тестів

Для тестування бізнес-логіки використано фреймворк xUnit та бібліотеку Moq для створення фіктивних (mock) репозиторіїв. Було написано 8 модульних тестів, які покривають як позитивні сценарії (успішна видача, реєстрація), так і негативні (спроба взяти позичену книгу, порожнє ім'я тощо).

Як запустити тести:

Переконайтеся, що у вас встановлено .NET SDK.

Відкрийте термінал у кореневій папці проєкту (там, де знаходиться файл .sln).

Виконайте команду:

- dotnet test

Або скористайтеся графічним інтерфейсом JetBrains Rider: перейдіть у вкладку Unit Tests та натисніть Run All (Ctrl + U, L).
